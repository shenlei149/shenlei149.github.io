[Lord of the io_uring] 第一节：Linux 异步编程简介 (完整原文翻译)我们生活在一个大多数应用都运行在云端的时代。每当用户连接到云应用时，大部分业务逻辑（通常运行在某种 Web 子系统中引入了大量的不确定性和性能开销。普通文件的问题 (The trouble with regular files)在一个不是非常繁忙的服务器上，读写文件可能不会花费很长时间。以我们上面提到的使用异步设计编写的 FTP 服务器为例。当它因大量并发用户同时下载和上传大量非常大的文件而变得非常繁忙时，作为程序员，你需要了解一个麻烦。在如此繁忙的服务器上，read(2) 和 write(2) 调用可能会开始大量阻塞。但 select(2)、poll(2) 或 epoll(7) 系列系统调用不能在这里帮助我们吗？不幸的是，不能。这些系统调用总是会告知普通文件已就绪可进行 I/O。这是它们的阿喀琉斯之踵。我们不会深入探讨其中的原因，但重要的是要理解，虽然它们对套接字效果很好，但它们对普通文件总是返回“就绪”。不幸的是，这使得文件描述符在异步编程下是不统一的。支持普通文件的文件描述符受到了歧视。出于这个原因，像 libuv 这样的库对普通文件的 I/O 使用了一个单独的线程池，暴露了一个向用户隐藏这种差异的 API。io_uring 中存在这个问题吗？ (Does this problem exist in io_uring?)不。 io_uring 无论是处理套接字还是处理普通文件，都提供了一个统一的接口。此外，由于 API 的设计，程序可以直接获取读取或写入文件描述符的数据，而不是先知道文件描述符何时就绪，随后再对其启动 I/O 操作，就像 poll(2) 或 epoll(7) 所做的那样。这并不是 io_uring 优于现有 Linux 异步 I/O API 的唯一优势。我们将在接下来的章节中讨论更多内容。脚注虽然在 Linux 下创建线程或进程都是通过 clone(2) 系统调用完成的，并产生相同的开销，但在创建之后，如果子进程（它共享父进程地址空间的只读副本）写入其页面，内核会为子进程创建父进程地址空间的副本，这是真正的开销。进程中的线程共享地址空间，因此不会产生这种复制开销。

[Lord of the io_uring] 第二节：什么是 io_uring？ (What is io_uring?)
io_uring 是由 Facebook 的 Jens Axboe 创建的 Linux 系统新异步 I/O API。它的目标是提供一个没有当前 select(2)、poll(2)、epoll(7) 或 aio(7) 系列系统调用局限性的 API，这些局限性我们在前一节中已经讨论过。考虑到异步编程模型的用户选择它首先是为了性能原因，那么拥有一个性能开销非常低的 API 是有意义的。我们将在后续章节中看到 io_uring 是如何实现这一点的。

io_uring 接口 (The io_uring interface)
io_uring 这个名字本身源于这样一个事实：该接口使用**环形缓冲区（Ring Buffers）**作为内核与用户空间通信的主要界面。虽然其中涉及系统调用，但它们被保持在最低限度，并且有一种你可以使用的轮询模式，以尽可能减少进行系统调用的需求。

有 2 个环形缓冲区，一个用于提交请求（提交队列，即 SQ），另一个用于通知你这些请求的完成情况（完成队列，即 CQ）。这些环形缓冲区在内核和用户空间之间共享。你通过 io_uring_setup() 设置这些缓冲区，然后通过两次 mmap(2) 调用将它们映射到用户空间。

你告诉 io_uring 你需要完成什么工作（读取或写入文件、接受客户端连接等），你可以将其描述为**提交队列条目（SQE）**的一部分，并将其添加到提交环形缓冲区的尾部（Tail）。

随后，你调用 io_uring_enter() 系统调用来告知内核有请求正在等待处理。可选地，io_uring_enter() 也可以等待内核处理完一定数量的请求后再返回，这样你就知道可以从完成队列中读取结果了。

内核处理提交的请求，并将**完成队列事件（CQE）**添加到完成队列环形缓冲区的尾部。你从完成队列环形缓冲区的头部（Head）读取 CQE。每个 SQE 对应一个 CQE，它包含该特定请求的状态。你可以根据需要继续添加 SQE 并收割（Reaping）CQE。

有一种轮询模式可用，在这种模式下，内核会轮询提交队列中的新条目。在这种模式下，当你向提交队列添加条目时，通常不需要调用 io_uring_enter() 来通知内核。

io_uring 的性能 (io_uring performance)
由于内核和用户空间之间存在共享的环形缓冲区，io_uring 可以成为一个**零拷贝（Zero-copy）**系统。当涉及在内核和用户空间之间传递数据的系统调用时，字节拷贝就变得必要了。但由于 io_uring 中的大部分通信是通过内核与用户空间共享的缓冲区进行的，这种巨大的性能开销被完全避免了。

在 Linux 下使用同步编程接口甚至异步编程接口时，每个请求的提交至少涉及一个系统调用。在 io_uring 中，你可以简单地通过添加多个描述你想要的 I/O 操作的 SQE 来添加多个请求，并只需对 io_uring_enter 进行一次调用。首先，这本身就是一个胜利。

但情况还会变得更好。你可以让内核进行轮询，并在你将 SQE 添加到提交队列时拾取它们进行处理。这避免了你为了告诉内核拾取 SQE 而必须进行的 io_uring_enter() 调用。对于高性能应用，这意味着更少的系统调用开销。有关更多细节，请参阅提交队列轮询（SQ Polling）教程。通过巧妙地利用共享环形缓冲区，io_uring 的性能实际上是受内存带宽限制的（Memory-bound），因为在轮询模式下，我们可以完全取消系统调用。

重要的是要记住，性能基准测试是一个具有某种共同参考点的相对过程。根据 io_uring 的论文，在参考机器上，在轮询模式下，io_uring 成功达到了 170 万个 4k IOPS，而 aio(7) 达到的是 60.8 万个。虽然超过了两倍，但这并不是一个公平的比较，因为 aio(7) 并不具备轮询模式。但即使禁用了轮询模式，io_uring 也能达到 120 万个 IOPS，接近 aio(7) 的两倍。要检查 io_uring 接口的原始吞吐量，有一种 no-op 请求类型。

[补充翻译]
虽然系统调用（我们已经习惯频繁地使用它们）看起来似乎不是什么显著的开销，但在高性能应用中，大量发起系统调用将产生重大影响。此外，系统调用的成本也不再像以前那样廉价。考虑到操作系统为了应对 Spectre（幽灵） 和 Meltdown（熔断） 漏洞而采取的变通（补丁）措施，我们讨论的是非同小可的开销。因此，在高性能应用中尽可能多地避免系统调用确实是一个极佳的主意。

底层接口简介 (Introduction to the low-level interface)
io_uring 的接口很简单。有一个提交队列（submission queue）和一个完成队列（completion queue）。在提交队列中，你提交关于想要完成的各种操作的信息。例如，在我们当前的程序中，我们想用 readv(2) 读取文件，所以我们放置一个提交队列请求，将其作为提交队列条目（SQE）的一部分进行描述。此外，你可以放置不止一个请求。队列深度（你可以定义）允许放置多少请求，你就可以放置多少。这些操作可以是读取、写入等的混合。然后，我们调用 io_uring_enter() 系统调用来告诉内核我们已经向提交队列添加了请求。接着内核会施展它的“柔术”（jujitsu），一旦处理完这些请求，它就会将结果作为 CQE（即完成队列条目）的一部分放入完成队列中，每个 CQE 对应一个 SQE。这些 CQE 可以从用户空间瞬间访问，因为它们被放置在一个由内核和用户空间共享的缓冲区中。

我们之前讨论过 io_uring 的这一特殊优势，但敏锐的读者会注意到，这种填满一个包含多个 I/O 请求的队列然后发起单次系统调用（而不是为每个 I/O 请求发起一次系统调用）的接口本身就已经更高效了。为了让效率更上一层楼，io_uring 支持一种模式，在这种模式下，**内核会轮询（poll）**你在提交队列中做出的条目，而你甚至不需要调用 io_uring_enter() 来通知内核有新的提交队列条目。另一点需要注意的是，在发现 Spectre 和 Meltdown 硬件漏洞以及操作系统为应对它们创建了变通方法之后，系统调用的成本比以往任何时候都要高。因此，对于高性能应用来说，减少系统调用的次数确实是一件大事。

在你做任何这些之前，你需要设置队列，它们实际上是具有一定深度/长度的环形缓冲区。你调用 io_uring_setup() 系统调用来完成这项工作。我们通过向环形缓冲区添加提交队列条目并从完成队列环形缓冲区读取完成队列条目来执行实际工作。这就是 io_uring 接口设计的一个概览。

完成队列条目 (Completion Queue Entry)
现在我们已经有了事情如何运作的心智模型，让我们更详细地看看这是如何做到的。与提交队列条目（SQE）相比，完成队列条目（CQE）非常简单。所以，让我们先看它。SQE 是 io_uring_sqe 结构的一个实例，你使用它提交请求。你将其添加到提交环形缓冲区。CQE 是 io_uring_cqe 结构的一个实例，内核对添加到提交队列的每个 io_uring_sqe 结构实例都会以该实例进行响应。它包含了你通过 SQE 实例请求的操作的结果。

C
struct io_uring_cqe {
  __u64  user_data;   /* sqe->user_data 提交时传递回来的数据 */
  __s32  res;         /* 此事件的结果代码 */
  __u32  flags;
};
将完成与提交相关联 (Correlating completions with submissions)
正如代码注释中所提到的，user_data 字段是从 SQE 原封不动传递到 CQE 实例的内容。假设你在提交队列中提交了一堆请求，它们不一定按相同的顺序完成并在完成队列中显示为 CQE。以以下场景为例：你的机器上有两块磁盘：一个是较慢的旋转硬盘，另一个是超快的 SSD。你在提交队列上提交了 2 个请求。第一个是在较慢的旋转硬盘上读取一个 100kB 的文件，第二个是在较快的 SSD 上读取一个相同大小的文件。如果必须维持顺序，即使来自 SSD 文件的预期数据能更快到达，内核是否应该等待来自旋转硬盘文件的数据变得可用？坏主意，因为这阻止了我们跑得尽可能快。所以，CQE 可以在它们变得可用时以任何顺序到达。无论哪个操作完成，其结果都会在 CQ 上可用。既然 CQE 到达没有指定的顺序，鉴于你现在知道 CQE 在上面的 io_uring_cqe 结构中长什么样，你如何识别某个特定的 CQE 对应哪个 SQE 请求呢？一种方法是使用 SQE 和 CQE 共有字段 user_data 来识别完成情况。并不是说你要设置一个唯一的 ID 之类的，你通常会传递一个指针。如果这令人困惑，请等一等，直到你稍后在这里看到一个清晰的例子。

完成队列条目很简单，因为它主要关注系统调用的返回值，该值在 res 字段中返回。例如，如果你排队了一个读取操作，在成功完成后，它将包含读取的字节数。如果有错误，它将包含一个负的错误编号。本质上就是 read(2) 系统调用本身会返回的内容。

顺序 (Ordering)
虽然我确实提到过 CQE 可以以任何顺序到达，但你可以通过 SQE 排序来强制某些操作的顺序，实际上是将它们链式连接起来。详情请参阅教程：链接请求（Linking requests）。

提交队列条目 (Submission Queue Entry)
提交队列条目比完成队列条目复杂一点，因为它需要足够通用，以表示和处理当今 Linux 可能进行的广泛 I/O 操作。

C
struct io_uring_sqe {
  __u8  opcode;   /* 此 sqe 的操作类型 */
  __u8  flags;    /* IOSQE_ 标志 */
  __u16  ioprio;  /* 请求的 ioprio */
  __s32  fd;      /* 执行 IO 的文件描述符 */
  __u64  off;     /* 文件偏移量 */
  __u64  addr;    /* 指向缓冲区或 iovecs 的指针 */
  __u32  len;     /* 缓冲区大小或 iovecs 的数量 */
  union {
    __kernel_rwf_t  rw_flags;
    __u32    fsync_flags;
    __u16    poll_events;
    __u32    sync_range_flags;
    __u32    msg_flags;
  };
  __u64  user_data;   /* 完成时传回的数据 */
  union {
    __u16  buf_index; /* 如果使用了固定缓冲区，则为索引 */
    __u64  __pad2[3];
  };
};
我知道这个结构体看起来很忙碌。常用的字段只有几个，通过像我们正在处理的 cat 这样简单的例子很容易解释：当你想要使用 readv(2) 系统调用读取文件时：

opcode 用于指定操作，在我们的例子中，是使用 IORING_OP_READV 常量的 readv(2)。

fd 用于指定代表你想要读取的文件的文件描述符。

addr 用于指向 iovec 结构体数组，这些结构体保存了我们为 I/O 分配的缓冲区的地址和长度。

最后，len 用于保存 iovec 结构体数组的长度。

现在这并不是太困难，不是吗？你填充这些值，让 io_uring 知道要做什么。你可以排队多个 SQE，并在你希望内核开始处理排队的请求时最终调用 io_uring_enter()。

初始设置 (The initial setup)
从 main() 中，我们调用 app_setup_uring()，它执行了我们使用 io_uring 所需的初始化工作。首先，我们调用 io_uring_setup() 系统调用，传入我们需要的队列深度和一个全部设为零的 io_uring_params 结构实例。当调用返回时，内核将填充该结构成员中的值。这就是 io_uring_params 的样子：

C
struct io_uring_params {
  __u32 sq_entries;
  __u32 cq_entries;
  __u32 flags;
  __u32 sq_thread_cpu;
  __u32 sq_thread_idle;

sq_ptr = mmap(0, sring_sz, PROT_READ | PROT_WRITE,

/* 映射提交队列条目数组 */
s->sqes = mmap(0, p.sq_entries * sizeof(struct io_uring_sqe),

        s->ring_fd, IORING_OFF_SQES);
我们将重要的细节保存在我们的结构 app_io_sq_ring 和 app_io_cq_ring 中，以便以后轻松参考。当我们分别为提交和完成映射两个环形缓冲区时，你可能会好奇第三个映射是干什么的。虽然完成队列环直接索引共享的 CQE 数组，但提交环在中间有一个间接数组（indirection array）。提交侧环形缓冲区是进入此数组的索引，而该数组反过来包含进入 SQE 的索引。这对于某些将提交请求嵌入内部数据结构中的应用很有用。这种设置允许它们一举提交多个提交条目，同时允许它们更容易地采用 io_uring。

【补充】第一次 mmap 除了 sring 的 tail head mask，还有 sring->array，指向最后一次 mmap 的 sqes 的具体位置。提交 entry 比较大，64B，分离。完成 entry 小，16B，内嵌。

注意：在内核版本 5.4 及以上，单个 mmap(2) 即可映射提交和完成队列。但在旧内核中，它们需要分别映射。你可以通过检查 IORING_FEAT_SINGLE_MMAP 特性标志来检查内核是否具有通过一次 mmap(2) 映射两个队列的能力，就像我们在上面的代码中所做的那样，而不是检查内核版本。

处理共享环形缓冲区 (Dealing with the shared ring buffers)
在常规编程中，我们习惯于处理用户空间和内核之间非常清晰的接口：系统调用。然而，系统调用确实是有代价的，对于像 io_uring 这样的高性能接口，希望尽可能地摆脱它们。我们早些时候看到，与我们通常进行的多次系统调用相比，使用 io_uring 允许我们批量处理许多 I/O 请求，并对 io_uring_enter() 系统调用进行单次调用。或者在轮询模式下，甚至不需要该调用。

从用户空间读取或更新共享环形缓冲区时，需要小心确保读取时看到的是最新数据，更新后要“刷新”或“同步”写入，以便内核看到你的更新。这是由于 CPU 可以重新排序读取和写入，编译器也可以。当读取和写入发生在同一个 CPU 上时，这通常不是问题。但在 io_uring 的情况下，当涉及跨越两个不同上下文（用户空间和内核）的共享缓冲区时，这些上下文可以在上下文切换后运行在不同的 CPU 上。你需要从用户空间确保在读取之前，之前的写入是可见的。或者当你填写 SQE 的细节并更新提交环形缓冲区的尾部（tail）时，你希望确保你对 SQE 成员所做的写入被排序在更新环形缓冲区尾部的写入之前。如果这些写入没有排序，内核可能会看到尾部已更新，但当它读取 SQE 时，它读取时可能找不到它需要的所有数据。在内核寻找尾部变化的轮询模式下，这变成了一个真正的问题。这都是因为 CPU 和编译器为了优化而重新排序读取和写入。

读取完成队列条目 (Reading a completion queue entry)
一如既往，我们首先处理完成方面的事情，因为它比提交方面更简单。之所以需要这些解释，是因为我们需要讨论内存排序（memory ordering）以及我们需要如何处理它。否则，我们只想看看如何处理环形缓冲区。对于完成事件，内核将 CQE 添加到环形缓冲区并更新尾部（tail），而我们在用户空间从头部（head）读取。在任何环形缓冲区中，如果头部和尾部相等，则表示环形缓冲区为空。看一看下面的代码：

C
unsigned head;
head = cqring->head;
read_barrier(); /* 确保之前的写入可见 */
if (head != cqring->tail) {
    /* 环形缓冲区中有可用数据 */
    struct io_uring_cqe *cqe;
    unsigned index;
    index = head & (cqring->mask);
    cqe = &cqring->cqes[index];
    /* 在此处理已完成的 cqe */
     ...
    /* 我们现在已经消耗了此条目 */
    head++;
}
cqring->head = head;
write_barrier();
为了获得头部的索引，应用程序需要用环形缓冲区的尺寸掩码（mask）对 head 进行掩码处理。记住，上面代码中的任何一行都可能在上下文切换后运行。因此，就在比较之前，我们有一个 read_barrier()，这样，如果内核确实更新了尾部，我们可以在 if 语句中将其作为比较的一部分进行读取。一旦我们获得 CQE 并对其进行处理，我们就更新头部，让内核知道我们已经消耗了环形缓冲区中的一个条目。最后的 write_barrier() 确保我们所做的写入变得可见，以便内核知道它。

进行提交 (Making a submission)
进行提交与读取完成相反。在处理完成时，内核将条目添加到尾部，我们从环形缓冲区的头部读取条目；而在进行提交时，我们添加到尾部，内核从提交环形缓冲区的头部读取条目。

C
struct io_uring_sqe *sqe;
unsigned tail, index;
tail = sqring->tail;
index = tail & (*sqring->ring_mask);
sqe = &sqring->sqes[index];
/* 此函数调用填充此 IO 请求的 SQE 细节 */
app_init_io(sqe);
/* 将 SQE 索引填充到 SQ 环数组中 */
sqring->array[index] = index;
tail++;
write_barrier();
sqring->tail = tail;
write_barrier();
在上面的代码片段中，应用程序中的 app_init_io() 函数填充了提交请求的细节。在尾部更新之前，我们有一个 write_barrier() 以确保之前的写入是有序的。然后我们更新尾部并再次调用 write_barrier() 以确保我们的更新被看到。我们在这里正“按部就班地准备”（lining up our ducks）。
